# RULES — Fase 4: Validator Stage

- **Sem código de negócio:** Proibido criar ou modificar código fonte nesta fase.
- **Validação estrita:** Execute fielmente e de forma determinística os comandos declarados no `build_report.md`.
- **Colete Logs:** Obtenha toda a saída do terminal (seja sucesso ou erro) para comprovar a veracidade do teste.
- **Auto-Correção Obrigatória:** Caso aconteça algum erro na fase 4 (validator), a execução NÃO PODE ser interrompida. Você deve registrar a falha temporária no `validation_report.md` e `state.json`, e prosseguir automaticamente para a próxima etapa apropriada (como assumir a Fase 5 de Review e retornar à Fase 3 de Builder) para corrigir o problema. Você deve manter o ciclo rodando de forma autônoma até que a validação passe com sucesso e o fluxo principal seja concluído.
- **Checklist de Validação Estrutural (Skills):** O Validator DEVE confrontar a entrega com os checks de qualidade estrutural, compatibilidade, estilo, performance e segurança correspondentes ao stack do projeto, conforme catalogado em [build-validation-checklist.md](file:///d:/projetos/AgentOrchestrix/skills/build-validation-checklist.md) e demais skills do stack, apontando quaisquer desvios como falhas estruturais de validação.
- **Validação de Restrições e Contratos:** O Validator DEVE verificar se os limites físicos e restrições acordadas na Fase 0 (ex: limite de linhas por arquivo) foram respeitados pelo código gerado, e garantir que todos os artefatos gerados estejam em conformidade com as estruturas H1/H2 definidas nos respectivos contratos.
- **Registro de Guardrails:** Ao concluir a validação, o Validator DEVE registrar no array `prevention_guardrails` do `state.json` quais validações da checklist de build foram executadas com sucesso.
- **Verificação de Conformidade Arquitetural:** O Validator DEVE comparar os nomes de arquivos, contextos, componentes e a estrutura de pastas declarados no `architecture.md` com o código real em `generated/`. Divergências entre o documento e a implementação (ex: nomes de contextos diferentes, subpastas inexistentes, componentes listados mas não criados) DEVEM ser registradas como falhas estruturais. O Validator deve utilizar `list_dir` e `grep_search` para essa verificação.
