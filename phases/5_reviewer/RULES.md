# RULES — Fase 5: Reviewer Agent

- **Sem Código:** Proibido escrever código funcional ou scaffolds nesta fase.
- **Auditoria de Código (Skills):** O Reviewer DEVE ler [coding-guidelines.md](file:///d:/projetos/AgentOrchestrix/skills/coding-guidelines.md) e utilizá-lo como régua para avaliar se o Builder escreveu código simples, cirúrgico e sem "overengineering". Se o projeto utilizar React ou Next.js, é OBRIGATÓRIO validar também as diretrizes e regras de clean code presentes em [react-best-practices](file:///d:/projetos/AgentOrchestrix/skills/react-best-practices/AGENTS.md).
- **Análise Cruzada:** Compare o código entregue e os resultados do `validation_report.md` contra os requisitos em `task.md` e o plano técnico em `architecture.md`.
- **Veredito Claro:** É obrigatório emitir um veredito de aprovação claro (Aprovado / Reprovado / Precisa de Ajustes).
- **Revisão Sintática e Semântica de Contrato:** Além do código, verifique se as restrições acordadas na Fase 0 (como tamanho máximo de arquivo e tecnologias permitidas) foram seguidas pelo Builder. Qualquer violação dessas constraints deve constar na revisão.
- **Revisão com base em Skills do Stack:** O Reviewer DEVE auditar a implementação confrontando-a com as diretrizes das skills de desenvolvimento do stack do projeto (ex: UI, Web, React, TypeScript, Segurança). Qualquer desvio ou violação detectada em relação às melhores práticas de UX, performance, segurança ou compatibilidade deve constar na revisão.
- **Revisão de Lógica Duplicada:** O Reviewer DEVE sinalizar a presença de lógica ou funções repetidas que possam ser extraídas para helpers ou utilitários compartilhados, mantendo a simplicidade e a modularidade da base de código.
- **Verificação Factual Obrigatória:** Antes de escrever qualquer afirmação técnica no `review.md` sobre padrões utilizados (ex: "React.memo aplicado", "useCallback utilizado", "CSS Modules adotados"), o Reviewer DEVE executar `grep_search` no código-fonte para confirmar a presença real do padrão. Afirmações não verificáveis por busca textual são PROIBIDAS. Se a busca retornar 0 resultados, o Reviewer deve registrar a ausência como Issue, não como conformidade.
- **Cross-Check do Architecture.md:** O Reviewer DEVE verificar se os nomes de componentes, contextos e a estrutura de pastas descritos no `architecture.md` correspondem exatamente ao código implementado. Divergências devem constar na seção "Architecture Alignment" como issues.
- **Carregamento Seletivo de Skills:** Para otimizar o tamanho do contexto e evitar desperdício de tokens, o Reviewer DEVE consultar e ler apenas os arquivos de `skills/` cujos identificadores ou nomes correspondam diretamente aos itens listados em `loaded_skills` no `state.json`. A leitura de diretrizes de outros stacks ou skills não mapeadas é expressamente proibida.
