# RULES — Fase 2: Architect Agent

- **Alinhamento Técnico:** Desenhe a arquitetura baseando-se estritamente no `task.md`.
- **Fim da Fase (Comportamento por Comando):** Se o comando for `orquestrar:`, altere a fase para `3_builder` no `state.json` e continue automaticamente a execução da implementação. Se o comando for `planejar:`, pare a execução, mude o status do `state.json` para `completed` e apresente os artefatos gerados.
- **Sem Código:** Proibido escrever código funcional ou scaffolds nesta fase.
- **Análise de Atrito de Restrições:** Ao desenhar a arquitetura, correlacione as decisões de escopo/tecnologia com as restrições (constraints) da Fase 0. Se houver restrições estritas de tamanho de arquivo (ex: <100 linhas), você é OBRIGADO a propor ferramentas de isolamento de estado (como Context API ou Zustand) para mitigar o acoplamento excessivo (prop drilling).
- **Design e Arquitetura:** Caso seja uma aplicação visual (Frontend/UI), o Architect DEVE ler [frontend-blueprint.md](file:///d:/projetos/AgentOrchestrix/skills/frontend-blueprint.md) e planejar atualizações atômicas do DOM (sem renderizações destrutivas).
- **React Best Practices:** Caso o projeto utilize React, o Architect DEVE consultar a skill [react-best-practices](file:///d:/projetos/AgentOrchestrix/skills/react-best-practices/SKILL.md) (e o consolidado em [AGENTS.md](file:///d:/projetos/AgentOrchestrix/skills/react-best-practices/AGENTS.md)) para planejar a estrutura de componentes considerando padrões de renderização/memoização, isolamento de estilos (CSS Modules) e acessibilidade (Focus Trap em modais).
- **Separação de Contextos de Estado:** Quando utilizar gerenciamento de estado global (como Context API ou Zustand), o Architect DEVE propor separação por domínio (ex: dados das tarefas vs. estado visual/modais vs. filtros) para evitar re-renderizações desnecessárias e acoplamento.
- **Checklist de Interações e Acessibilidade:** O Architect DEVE mapear no design todas as interações complexas planejadas (ex: drag-and-drop de itens, reordenação de containers, menus suspensos) e especificar a biblioteca acessível ou o fallback programático para teclado e touch.
- **Avaliação de Limites de Storage:** Caso o armazenamento seja local (client-side), o Architect DEVE avaliar o impacto dos limites físicos (ex: 5MB do localStorage), propor estratégias de migração ou degradação e planejar um fluxo de exportação/backup.
